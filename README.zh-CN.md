# Layered Web Access

[English README](README.md)

面向 `Codex` 和 `Claude Code` 的安全优先网页访问 Skill。  
当前版本：`0.1.0-alpha.1`

## 这个项目是什么

`layered-web-access` 不是“全能浏览器接管器”，而是一个**分层、可解释、风险受控**的网页访问产品。

它的目标是：

- 优先完成公开网页研究和内容提取
- 优先走更低风险的访问路径
- 只在确实必要时，才进入显式开启的高级模式
- 让能力边界、风险边界、支持边界都保持清楚

它适合的任务包括：

- 公开网页研究
- 已知 URL 内容提取
- 高价值公开技术内容访问
- 一部分内容平台详情页提取
- 有限动态页的受控读取

它**不**是一个以“默认拿最高权限浏览器能力”为目标的产品。

## 这个项目的核心特点

### 1. 安全优先

- 默认不走高权限浏览器路径
- 优先使用搜索、静态抓取和站点专用提取器
- 不把高风险能力放进默认态

### 2. 双模式

- `默认模式`
  - 面向大多数公开网页任务
  - 优先走低风险、低成本路径
- `高级模式`
  - 必须由用户显式开启
  - 才会进入隔离浏览器 profile 的增强路径

### 3. 公开内容优先

- 正式支持目标是公开内容和有限动态页
- 不把登录后读取当作第一版正式承诺
- 不把 OAuth、账号态、挑战站点强攻当作默认能力

### 4. 边界清楚

这个项目强调：

- 支持什么，要明确写出来
- 不支持什么，也要明确写出来
- 实验性支持与正式支持必须分开

## 产品决策：为什么这样设计

这个项目最重要的产品决策是：

- **不把“最高权限浏览器能力”当作默认能力**

很多网页访问工具为了提升成功率，会更倾向于：

- 接管用户日常浏览器
- 复用已有登录态
- 尽早开放更强浏览器控制能力
- 在 challenge / 登录墙 / OAuth 上继续强攻

这样做的优点是：

- 在一部分高挑战站点上，上限可能更高

但代价同样明显：

- 风险更高
- 损害半径更大
- 用户更难理解 Agent 到底掌握了什么权限
- 一旦误用，更接近“Agent 在替你本人操作浏览器”

`layered-web-access` 刻意选择了另一条路线：

- 先用最低风险路径完成公开网页任务
- 优先走搜索、静态抓取和站点专用提取器
- 只在确实需要时，再进入显式开启的高级模式
- 高级模式也只面向公开内容和有限动态页

一句话总结：

- **默认更安全，必要时再变强，而不是默认最强**

## 当前不能做到什么，为什么

当前版本**不正式承诺**以下能力：

- 登录后网站读取
- OAuth 登录流程
- 稳定访问 `x.com`
- 稳定访问 `platform.openai.com`
- 直接接管用户日常浏览器 profile

原因不是“没想过”，而是明确的产品选择：

- 登录后读取意味着权限边界很快上升到账户级风险
- OAuth 和高挑战站点往往把问题从“网页读取”变成“浏览器身份与风控信任”
- 接管日常浏览器虽然可能提高成功率，但会显著扩大隐私与误操作风险

所以这里不是简单的“暂时没做”，而是：

- **第一版故意不把这些能力纳入正式承诺**

## 为什么说它更安全

这里的“更安全”，不是零风险，而是：

- 高风险能力不在默认态
- 浏览器路径要求显式进入高级模式
- 使用隔离 profile，而不是用户日常 profile
- 正式支持范围限定为公开内容
- 站点边界写清楚，不靠模糊承诺掩盖风险

这会带来一个现实结果：

- 它的能力上限，确实不会等同于一些高权限浏览器自动化工具
- 但默认风险、误用风险和损害半径，也会明显更低

## 实际效果是什么

在当前产品边界下，这个项目的实际效果是：

- 可以较稳定地处理高价值公开内容
- 可以把一部分常见网页任务从“纯浏览器操作”降到“可维护的提取路径”
- 可以把默认使用体验保持在更清楚、更可解释的风险范围内

它追求的不是：

- “任何站点都强行打通”

而是：

- **让公开网页访问这件事，尽量可用、可解释、可维护、可发布**

## 项目背景

这个项目不是凭空设计出来的，而是在评估多种现有方案之后，主动收敛出来的产品路线。

主要参考方向包括：

- [`eze-is/web-access`](https://github.com/eze-is/web-access)
  - 提醒我们：任务路由很重要，应该先走简单路径，再走强能力路径
- [`epiral/bb-browser`](https://github.com/epiral/bb-browser)
  - 提醒我们：隔离 profile、结构化浏览器控制、浏览器能力分层都很重要
- [`teng-lin/agent-fetch`](https://github.com/teng-lin/agent-fetch)
  - 提醒我们：在静态抓取和浏览器控制之间，还存在更强 HTTP 抓取这一中间层

但这个项目**并不是对这些仓库的直接移植**。

它做的是另一种产品选择：

- 更安全的默认值
- 更明确的模式边界
- 更克制的正式支持范围
- 更低的默认风险

这意味着：

- 它的能力上限不会和某些高权限工具完全一样
- 但默认风险也更低，更适合公开发布和长期维护

## 当前支持范围

第一版正式支持重点：

- 公开内容
- 静态提取
- 站点专用提取器
- 隔离 profile 下的有限动态页

当前已覆盖的高价值站点家族包括：

- Reddit 公共 feed 和 thread
- Stack Overflow 与部分 Stack Exchange 站点
- arXiv
- Hugging Face
- 部分官方文档站
- YouTube watch 页面
- BBC 文章页
- 微信文章页
- Wikipedia 页面

实验性支持：

- Bilibili 公共视频页

## 当前明确不承诺的头部站点 / 站点类型

以下站点或页面类型目前在正式支持边界之外，或已经测试后明确延期：

- `x.com`
  - challenge / 登录墙较重，不属于第一版正式目标
- `platform.openai.com`
  - challenge-heavy，且与 OAuth / 浏览器信任问题交织，不属于第一版正式目标
- `zhihu.com/question`
  - 已测试，但当前公开路径没有稳定拿到可用正文
- `reuters.com` 文章页
  - 已测试，但当前公开路径没有稳定拿到可用正文

这些内容写出来不是为了“回避问题”，而是为了让边界保持诚实。

## 模式说明

### 默认模式

- 安全优先
- 优先使用宿主原生 web/search 能力、静态抓取、站点专用提取器
- 不应把 Agent 临场安装依赖作为常规路径

### 高级模式

- 必须由用户显式开启
- 使用隔离浏览器 profile
- 仍然只面向公开内容和有限动态页
- 当本地高级模式依赖缺失时，允许引导式准备
